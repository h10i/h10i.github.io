WSL で Windows の PATH を引き継がないようにする

2025-09-11 · 2 min · Me | Suggest Changes
Table of Contents
  • 概要
  • 環境

  • 調査
  • 対処

概要#

  • WSL は、デフォルトで Windows の PATH を引き継いでいる。
  • Windows と WSL 上でパスが競合すると、意図しない方が呼び出されてしまう。
  • WSL が Windows の PATH を引き継がないように設定する。

環境#

  • Windows 11
  • WSL 2.3.24.0

事象#

Windows 11 に Gemini CLI をインストールし、WSL 上でも Gemini CLI をインストールした。

Windows 11 上で使う Gemini CLI は gemini コマンド実行後直ぐに開始する。しかし、WSL 上の Gemini CLI は gemini 実行後、開始までに数十秒かかる。

調査#

gemini コマンドの場所を調べると、/mnt/c/... と表示された。

which gemini

WSL は Windows の PATH をデフォルトで引き継ぐ設定となっている。そのため、WSL 上で cmd.exe や notepad.exe を実行すると、そのまま Windows のアプリとして実行できる。

この設定により、WSL 上にある gemini ではなく、Windows 11 上にある gemini を WSL 上から呼び出していた。WSL から Windows のファイルを読み書きするのは時間がかかるため、gemini の開始が遅くなっていた。

対処#

WSL の設定で Windows の PATH を引き継がないようにする。

WSL にて、wsl.conf を開く。

sudo vi /etc/wsl.conf

以下を追記し、上書き保存する。

[interop]
appendWindowsPath=false

Windows の PowerShell に戻り、WSL を再起動する。

wsl.exe --shutdown

再度 WSL に接続したところ、Windows の PATH は含まれていない状態になっていた。

しかし、gemini を実行しても、Command 'gemini' not found になった。そもそも PATH を引き継いでいたため WSL 側にはインストールできておらず、Windows 側にインストールされていたものと思われる。

再度 gemini をインストールしたところ、WSL 上にインストールされた。また、コマンド実行直後に開始するようになった。
